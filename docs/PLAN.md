# teledex 项目计划

## 1. 产品目标

`teledex` 的目标是做一个通用、轻量的 Telegram 控制 Codex 工具。

它要解决的问题很明确：

- 用户希望在 Telegram 里直接驱动 Codex
- 用户希望一个会话可以持续，不要每次重来
- 用户希望不同项目目录之间可以切换
- 用户希望执行过程可见，而不是黑盒等待
- 用户希望出现异常时可以快速恢复或重试

第一版不追求复杂平台化，不引入桌面管理面、工作区托管运行时、平台强绑定设计，只把最核心的链路做扎实。

## 2. 产品定义

一句话定义：

`teledex` 是一个通过 Telegram 远程控制 Codex、支持会话持续和目录绑定、并能展示执行过程预览的通用桥接服务。

## 3. 第一版核心功能

### 3.1 Telegram 控制入口

- 用户通过 Telegram 和 bot 交互
- 支持命令与普通消息两种输入形式
- 普通消息默认发往当前活跃会话

### 3.2 用户鉴权

- 只允许白名单用户调用
- 未授权用户收到明确拒绝提示

### 3.3 会话管理

- 新建会话
- 查看会话列表
- 切换当前会话
- 继续当前会话

### 3.4 工作目录绑定

- 每个会话可以绑定一个工作目录
- 执行时默认在该目录中运行
- 用户可查询当前会话绑定目录

### 3.5 Draft 过程预览

- 收到任务后先返回一条“处理中”过程消息
- 执行过程中持续刷新同一条消息
- 展示当前状态或阶段性文本
- 任务结束后停止更新

### 3.6 最终结果回传

- 任务完成后发送正式结果
- 长结果支持截断、分段或附件回退

### 3.7 错误反馈

- 目录不存在
- 会话不存在
- Codex 调用失败
- 正在执行时重复触发
- Telegram 发送失败

这些错误都要给出简洁、明确、可操作的提示。

## 4. 第一版命令设计

- `/start`：初始化说明与权限校验
- `/new`：新建会话
- `/sessions`：查看会话列表
- `/use <id>`：切换当前会话
- `/bind <path>`：为当前会话绑定工作目录
- `/pwd`：查看当前会话绑定目录
- `/stop`：停止当前任务

普通文本消息：

- 默认发送给当前活跃会话
- 在当前绑定目录中执行
- 自动进入过程预览与最终结果返回流程

## 5. 项目主模型

整个项目围绕四个对象展开：

### 5.1 User

表示 Telegram 使用者，维护：

- 用户 ID
- 是否授权
- 当前活跃会话
- 最近活跃时间

### 5.2 Session

表示一个可持续的 Codex 会话，维护：

- 会话 ID
- 标题
- 当前状态
- 绑定目录
- 最近活跃时间

### 5.3 Workspace

这里不做复杂平台托管，只表示“当前会话绑定的真实目录”。

### 5.4 Run

表示一次具体执行任务，维护：

- 运行 ID
- 关联会话
- 当前状态
- 过程消息映射
- 开始与结束时间
- 错误信息

## 6. 实现思路

虽然当前不写代码，但后续实现建议按以下结构推进：

### 6.1 Telegram Adapter

职责：

- 接收 Telegram 更新
- 解析命令
- 发送普通消息
- 更新过程消息
- 发送最终结果

### 6.2 Session Manager

职责：

- 创建会话
- 获取当前会话
- 切换会话
- 列表展示
- 绑定目录
- 查询当前目录

### 6.3 Codex Runner

职责：

- 新建 Codex 会话
- 恢复已有会话
- 把用户输入发送给 Codex
- 读取过程输出
- 提取最终回复

### 6.4 Draft Controller

职责：

- 创建一条过程消息
- 按节流策略更新
- 避免重复内容重复刷新
- 任务结束后停更

### 6.5 Storage

职责：

- 保存用户状态
- 保存会话信息
- 保存目录绑定
- 保存运行状态
- 支持进程重启后恢复上下文

## 7. Draft 预览策略

这是第一版的关键体验能力，建议作为一等功能来实现。

预览逻辑建议如下：

1. 收到任务后，立即创建一条过程消息
2. 初始状态显示 `Preparing...`
3. 有过程文本时，用阶段性文本覆盖状态
4. 没有过程文本时，显示简短状态：
   - `Preparing...`
   - `Thinking...`
   - `Using tools...`
   - `Finalizing...`
5. 对更新频率做节流，避免 Telegram 刷屏
6. 最终结果返回后，过程消息停止更新

## 8. 数据落地建议

第一版建议使用轻量持久化。

建议至少保存这些内容：

- 用户当前活跃会话
- 会话基本信息
- 会话绑定目录
- 当前运行状态
- 过程消息 ID

目标只有一个：重启后不要把使用状态丢掉。

## 9. 开发顺序建议

如果后续开始实现，建议顺序如下：

1. 先定命令与数据模型
2. 先写存储层
3. 再写会话管理
4. 再写 Codex Runner
5. 再写 Draft 过程预览
6. 最后接 Telegram Adapter
7. 最后补文档和自测

这个顺序的目的，是先把“状态”和“执行”两条主链确定，再接外部交互层。

## 10. MVP 验收标准

第一版只要满足这些条件，就算可用：

- 白名单用户能和 bot 正常交互
- 能新建并继续一个会话
- 能切换会话
- 能绑定并查看工作目录
- 发普通消息后能触发 Codex
- 执行中能看到过程预览
- 完成后能收到最终结果
- 常见错误能收到明确提示

## 11. 暂不纳入第一版的内容

以下内容先不做：

- 桌面管理界面
- 多平台适配器
- 复杂工作区托管
- 会话归档系统
- 图片输入和文件上传
- 任务队列
- 复杂权限分级
- 运行时健康大盘

这些能力可以以后按需增加，但不进入第一版边界。

## 12. 总结

`teledex` 的第一版应该坚持一个原则：

先把“Telegram 中稳定控制 Codex 会话”这件事做好，再考虑更重的平台能力。

也就是说，第一版最重要的不是功能数量，而是这条主链是否顺畅：

`发消息 -> 找到当前会话 -> 在绑定目录里执行 -> 展示过程预览 -> 回传最终结果`
